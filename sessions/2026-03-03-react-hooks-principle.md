## 问答 1：Hooks 设计动机

### 导师提问

**React 为什么要引入 Hooks？它解决了 Class 组件的什么本质问题？**

你可以从以下几个角度思考：

1. **逻辑复用**：Class 组件时代我们用什么复用逻辑？有什么痛点？
2. **代码组织**：Class 组件中，相关逻辑（比如订阅、事件监听）通常写在哪里？有什么问题？
3. **代码压缩**：Class 的 minified 代码体积和函数组件相比如何？
4. **学习曲线**：`this` 关键字对初学者友好吗？

### 学员回答

react hooks 解决了 class 同一个状态不同时期调用的状态问题，react hooks 通过函数式编程闭包实现当前状态快照，而 class 是通过对象引用去获取当前状态。

class 组件中的相关逻辑都是写 react class 生命周期中的，需要管理 react 生命周期。

### 导师解答

**1. 逻辑复用的痛点**

Class 组件时代的方案：

**高阶组件（HOC）问题**：
- 嵌套层级过深，产生"包装地狱"
- 调试困难：DevTools 中看到多层 HOC 包裹
- 静态方法丢失：需要手动转发
- props 冲突：多个 HOC 可能注入同名 props

**Render Props 问题**：
- 嵌套过深，代码呈金字塔结构
- 无法在条件语句中使用：只能在 JSX 中嵌套

**问题本质**：
- 这些模式都需要改变组件结构来复用逻辑
- 逻辑被拆散到不同层级，难以追踪

**Hooks 的解决方案**：
- 逻辑按功能组织：相关的状态和副作用写在一起
- 不改变组件结构：直接调用自定义 Hook
- 组合优于继承：多个 Hook 可以线性组合

---

**2. 代码组织的混乱**

Class 组件的生命周期方法局限：

一个订阅功能在 Class 组件中需要拆散到三个地方：
1. **componentDidMount**：建立订阅
2. **componentDidUpdate**：props 变化时更新订阅
3. **componentWillUnmount**：清理订阅

**问题**：
- 相关逻辑被物理分离到不同方法
- 容易遗忘清理，导致内存泄漏
- 多个订阅时，代码重复且难以维护

**Hooks 的解决方案**：
- useEffect 统一处理：建立订阅、更新订阅、清理订阅写在同一个地方
- 依赖数组自动判断何时需要更新
- 清理函数自动在卸载时执行

---

**3. 状态一致性问题**

Class 组件的状态问题 - 对象引用的陷阱：

Class 组件中，`this.state` 是一个共享的可变对象引用。

场景：异步回调中读取状态
1. 用户点击按钮，触发异步请求
2. 请求发出后，用户又更新了状态
3. 请求返回时，回调读取 `this.state`
4. 读到的是最新状态，不是请求发出时的状态

**问题本质**：
- 异步回调执行时，无法确定状态是什么
- 需要手动保存"快照"（比如用变量暂存）
- 容易出错，需要额外的心智负担

**Hooks 的解决方案 - 函数式组件的状态快照**：

每次渲染都有独立的 props 和 state：
- 第一次渲染：props=A, state=1
- 第二次渲染：props=B, state=2

每次渲染的函数都是新的闭包：
- 闭包捕获的是当次渲染的 props/state
- 异步回调读到的是调用时的值，不会被后续更新影响

---

**4. this 关键字的困扰**

Class 组件的 this 问题：
- JavaScript 的 `this` 本身就是难点
- 需要手动绑定：`this.handleClick = this.handleClick.bind(this)`
- 箭头函数可以解决，但又涉及性能权衡

**常见错误**：
- 忘记绑定 this：`TypeError: Cannot read property 'state' of undefined`
- this 指向错误：回调中 this 不是组件实例
- 静态方法中无法访问 this

**Hooks 的解决方案**：
- 纯函数，无 this
- 直接使用 props/state，没有指向问题
- 代码更简洁，压缩率更高

---

**5. 总结对比**

| 维度 | Class 组件 | Hooks |
|------|-----------|-------|
| 逻辑复用 | HOC/Render Props（嵌套深） | 自定义 Hook（线性组合） |
| 代码组织 | 拆散到生命周期 | 按功能内聚 |
| 状态一致性 | 对象引用（共享可变） | 闭包快照（独立不可变） |
| this 问题 | 需要手动绑定 | 不需要 |
| 代码体积 | 较大 | 较小 |
| 学习曲线 | 陡峭 | 平缓 |

---

**6. Hooks 的设计哲学**

核心思想：
1. **函数优先**：用函数表达组件，而非类
2. **组合优于继承**：通过组合 Hook 构建复杂逻辑
3. **约定优于配置**：通过命名约定（use 开头）简化规则
4. **时间维度清晰**：每次渲染都有独立的状态快照

设计约束：
- 必须在顶层调用（保证顺序稳定）
- 只能在函数组件或自定义 Hook 中调用（限制使用场景）
